# redundant_assignment's VariableTracker keeps its own scope model

ADR 0001's architecture review named `VariableTracker` in `redundant_assignment/analysis.py` as a third hand-rolled scope walker duplicating `forbid_vars`/`validate_function_name`'s logic, both of which were later migrated onto the shared `_scope.iter_within_scope`/`SCOPE_NODES`. `VariableTracker` is not a duplicate of the same thing, though: it only opens a new `scope_id` for `FunctionDef`/`AsyncFunctionDef`/`ClassDef` — comprehensions and lambdas are deliberately tracked as a `comprehension_depth` flag on the _enclosing_ scope's assignments/uses, not as their own scope, because the redundant-assignment analysis cares about "is this comprehension result being assigned redundantly," not about modeling comprehension-local variable shadowing precisely. `_scope.py`'s `SCOPE_NODES` includes comprehensions and lambdas as hard boundaries (correct for the rename/collision-avoidance use cases in `forbid_vars` and `validate_function_name`), so routing `VariableTracker` through it would silently change which statements get grouped into which scope for TRI005's own detection logic — a real behavior change to a check that works well in practice, not a pure refactor.

`VariableTracker` is also structurally a poor fit: it's a single full-file `ast.NodeVisitor` pass that must descend into every scope while tagging `scope_id` as it goes, whereas `iter_within_scope`'s "yield a boundary node but never descend into it" shape assumes a caller that wants to stop at a boundary. There's no isolated "which node types are boundaries" table to extract from `VariableTracker` either, since its boundary decision is expressed through separate `visit_FunctionDef`/`visit_ClassDef` method overrides rather than a shared classification.

## Decision

Leave `VariableTracker`'s scope tracking as its own, narrower implementation. Consolidating it onto `_scope.py` was the wrong abstraction for what it does — the two model different things (comprehension-local shadowing precision vs. depth-tagged redundancy grouping), not the same thing implemented twice.
