# atomic_write_text() validates a fix's output before writing it

Every check's `fix()` builds a full new source string and writes it through the shared `atomic_write_text()` (`_base.py`), which was already atomic against I/O failures (temp-file-then-rename, per `docs/adr/0004-encoding-newline-fidelity.md`) but did nothing to validate the _content_ being written. If a bug in a check's fix logic ever produced syntactically invalid output, `atomic_write_text()` would write it anyway — the original, valid content is gone the moment the rename succeeds, with no re-parse anywhere in the write path to catch it. Today's actual defense against this is conservative heuristics that refuse to mark something fixable in the first place (e.g. `redundant_assignment` excluding augmented-assignment targets, `validate_function_name`'s `should_autofix` refusing methods/large/complex functions) — fix-then-validate, not just avoid-when-risky.

## Considered Options

- **Validate inside each check's own `fix()`**: rejected — would duplicate an `ast.parse()`-and-decide dance across all 5 checks with real fix logic, and a new check could easily forget it.
- **Validate in `CheckOrchestrator._apply_fixes`, wrapping each `check.fix()` call**: rejected — `atomic_write_text()` already commits the write (via `os.replace()`) before `fix()` returns, so validating after the call means briefly writing bad content to the real file before the orchestrator notices and reverts it. A crash in that window would leave the broken version behind, and "revert" would need its own read-before/write-back-after logic.
- **Validate inside `atomic_write_text()` before the rename, reject via exception**: adopted. `atomic_write_text()` is already the single chokepoint every fixable check writes through (confirmed: `forbid_vars`, `excessive_blank_lines`, `misplaced_comment`, `redundant_assignment`, `validate_function_name` all call it directly; `redundant_super_init` has no autofix at all). Since it already builds the new content in a temp file before renaming, validating that content first means a bad fix never gets far enough to create the temp file, let alone rename it into place — "revert" becomes "never commit," not "write then undo." One new `FixValidationError(path, syntax_error)` exception, raised before any file I/O, is the only change needed at this layer.
  - Validation uses `compile(content, str(path), "exec")`, not `ast.parse()`: some invalid code is only rejected at compile time, not by the grammar alone — `return`/`yield` outside a function, `break`/`continue` outside a loop, a `from __future__ import` that isn't the first statement. All still raise `SyntaxError`, just later in the pipeline than parsing; `ast.parse()` alone would let a fix producing e.g. bare `return` at module scope through.

## Consequences

- `_base.py` gains `FixValidationError` and a `mark_fix_rejected()`/`is_fix_rejected()` pair mirroring the existing `mark_fixed()`/`is_fixed()` convention (`Violation.fix_data["fix_rejected"]`).
- A check with a single write per `fix()` call (`forbid_vars`, `excessive_blank_lines`, `misplaced_comment`, `redundant_assignment`) needs no change: none of them catch a broad `Exception` around their write, so `FixValidationError` propagates uncaught out of `fix()`, and `CheckOrchestrator._apply_fixes` attributes the rejection to every violation that check reported for the file (accurate, since these checks are all-or-nothing — one combined write per call).
- `validate_function_name` is the one check that writes once per violation, in a loop, re-reading the file between each rename (each rename can shift line/col positions for the next). Its `fix()` now catches `FixValidationError` around each individual `apply_fix()` call and calls `mark_fix_rejected()` on that specific violation, then continues the loop — an earlier rename that already committed stays; a later one is still attempted. No snapshot/rollback machinery, no `ASTCheck.fix()` protocol change.
- One check's rejected fix doesn't block another check's unrelated fix on the same file, matching the pre-existing per-check `try`/`except` isolation in `_apply_fixes` — nothing new needed there either.
- **Drive-by fix, discovered while implementing the above**: `_apply_fixes` used to mark _every_ violation of a check as fixed whenever `fix()` returned `True` at all, regardless of which specific violation was actually touched. This was already wrong before this change — `validate_function_name`'s `should_autofix` can skip a method rename while fixing a free function in the same call, and the skipped one was reported `[FIXED]` anyway. Precisely attributing a rejection to the one violation that caused it (rather than the whole check) requires the same fix: `_apply_fixes` now re-reads and re-checks the file _after_ calling `fix()`, and compares each fresh violation's `(line, col, message)` against what's still present, instead of trusting the check's own boolean return.
  - The comparison is against `fresh_violations` (recomputed immediately before `fix()` ran), not the outer, long-stale `violations` list collected once at the very start of the file's processing — matching a stale entry back to "is this the same violation, now fixed" isn't reliable, since an earlier check's own fix can shift line/col numbers, and two distinct violations can share an identical message (e.g. a same-named free function and method both suggesting the same rename — confirmed reproducible, covered by a regression test). Rather than invent a more elaborate matching key, the stale entries for a check_id being processed are discarded outright once fresh, correctly-marked ones exist to replace them — `fresh_violations`' positions are current as of just before `fix()` ran, strictly more accurate than the original snapshot regardless.
  - This is a strict correctness improvement with no protocol change, done in the same pass since it shares the exact mechanism as the rejection-marking above.
- `main()`'s reporting loop gains a `[FIX REJECTED]` tag, shown instead of `[FIXABLE]`/`[FIXED]`, with a message pointing at the issue tracker instead of "Run with --fix" (re-running would fail identically).
