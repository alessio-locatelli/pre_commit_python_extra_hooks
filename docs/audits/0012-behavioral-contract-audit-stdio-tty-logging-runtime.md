# Investigation report: behavioral contract audit, stdin/stdout/TTY, logging & debugging, installation and runtime environment (ch. 26, 27, 29)

Detailed findings behind `docs/adr/0022-behavioral-contract-audit-stdio-tty-logging-runtime.md`. `docs/behavioral_contract.md` chapters **26** (Standard Input, Output, and TTY Behavior), **27** (Logging and Debugging), and **29** (Installation and Runtime Environment) were checked line-by-line against `_cli.py`'s argument parsing and `main()`, `__main__.py`'s process entry point, `_diagnostics.py`'s report formatting, every module's own `logging.getLogger(...)` usage, `_cache.py`'s `fcntl` handling, `_discovery.py`'s/`_prefilter.py`'s `git` subprocess fallbacks, and `pyproject.toml`'s dependency/version metadata, following on from `0011`–`0021`'s prior audits, per issue #38.

Grepped the whole source tree for every pattern each chapter could plausibly touch: `stdin`, `isatty`, `tty`/`TTY`, `input(`, process-group control, ANSI/color escape codes, `basicConfig`/`setLevel`/`addHandler`/`isEnabledFor`, and `os.environ`/`getenv` — every one of those came back empty except the logging-related search. `pyproject.toml`'s `[project]` table has no `dependencies` key at all — this project has zero runtime dependencies beyond the standard library — and `0021` already established there's no plugin mechanism.

## Already satisfied, confirmed with citations

- **Ch. 26, every item**: this tool never reads from, writes to, or queries a terminal in any TTY-aware way — no `stdin` read path exists at all (a literal `-` argument is treated as an ordinary, nonexistent file and cleanly reported, verified with a live `uv run ... - < /dev/null` invocation: exit 1, no hang), no `isatty()`/`tty`/`termios`/`curses` call and no ANSI/color escape code anywhere, and no `input()` prompt or foreground-process-group assumption. Every diagnostic is a plain `print(..., file=sys.stderr)` line, identical regardless of whether stdout/stderr are a TTY, a pipe, a redirected file, or a CI log collector.
- **Ch. 27, "MUST make it possible to identify the relevant file, rule, and operation during internal failures"**: already true without any log-level change — `_diagnostics.py`'s normal-output report names the exact file and `check_id` for both an unprocessable file and a rule crash.
- **Ch. 27, "MUST NOT log secrets or sensitive source content unnecessarily"; "MUST avoid logging enormous source contents by default"**: every `logger.*(...)` call site logs only a `filepath`, a `check_id`, or `repr(error)`; none references a `source`/`content` variable holding actual file text.
- **Ch. 27, "MUST ensure that debug logging does not change lint results"; "does not change auto-fix behavior"**: no call site anywhere branches on `logger.isEnabledFor()`/`getEffectiveLevel()` — every `logger.debug(...)` call is a pure side effect with no return value consulted by check/fix logic.
- **Ch. 29, "MUST provide a clear error when required runtime dependencies are unavailable"**: there are no required runtime dependencies to be unavailable — every import across `src/` is either always-present stdlib or `fcntl`, the one POSIX-only stdlib module, already handled (`0020`).
- **Ch. 29, "MUST handle missing optional dependencies gracefully"**: satisfied by two pre-existing, independent fallbacks — `_cache.py`'s `import fcntl` (`0020`) degrades to a disabled-but-correct cache with one explicit warning; `_discovery.py`'s/`_prefilter.py`'s `git` calls fall back to an equivalent pure-Python scan, logged only at `DEBUG` since that fallback produces the identical candidate set.

## Gap found and fixed

- **Chapter 27's "MUST provide a debug or verbose mode when normal output is insufficient for troubleshooting" had no way to reach it through this tool's own CLI.** Every module already logs the real exception behind a reported failure via `logger.debug(..., exc_info=True)` (`0017`'s logging fix deliberately downgraded these to `DEBUG` to stop duplicating the clean diagnostic line `_diagnostics.py` already prints) — but nothing in this codebase ever calls `logging.basicConfig()`, and no CLI flag existed to raise the level either. A user who hit a rule-crash or unreadable-file message had no supported way to see why: not via `.pre-commit-hooks.yaml`'s plain `entry: python -m pre_commit_hooks.ast_checks` invocation, and not via any documented flag or environment variable. Fixed by adding `-v`/`--verbose` to `_cli.py`'s argument parser (mirroring `ruff`'s own `-v`/`--verbose` flag, consistent with `0008`'s CLI-parity design): when set, `main()` calls `logging.basicConfig(level=logging.DEBUG, stream=sys.stderr)` once, before any file processing, surfacing every already-existing debug-level log line for the rest of the run. No other behavior changes: the flag only reconfigures logging, verified (not just asserted) by `test_verbose_flag_does_not_change_violations_or_exit_code`, which diffs a `--verbose` and a non-`--verbose` subprocess run of the same violation byte-for-byte on exit code and the violation line itself. Regression tests: `test_verbose_flag_surfaces_the_underlying_exception_on_stderr` and `test_verbose_flag_does_not_change_violations_or_exit_code` (`tests/test_main.py`), both real subprocess invocations — matching `0017`'s own `test_real_invocation_does_not_leak_a_traceback_onto_stderr` pattern, since pytest's own logging-capture plugin masks `logging.basicConfig()`'s real no-configuration-yet default from any in-process call. `test_main_verbose_flag_does_not_change_a_clean_runs_exit_code` (`tests/test_orchestrator.py`) additionally exercises the new branch in-process, needed only for coverage — the two subprocess tests above are the actual behavioral proof.

## Items judged not applicable, with rationale

- **Ch. 29, "MUST report incompatible runtime versions clearly"**: not implemented as an in-code runtime check, deliberately. This project's own source already uses syntax that doesn't exist before Python 3.14 (PEP 758, `0021`) in its own entry-point module, so an incompatible interpreter fails to even compile `__main__.py` before any runtime check written in this codebase could run. The only way to place such a check ahead of the failure would be a second shim module deliberately written in a syntax subset compatible with much older Pythons — exactly the kind of compatibility branch `AGENTS.md` explicitly forbids. In practice this is already caught one layer up: `pip`/`uv` both read `requires-python = ">=3.14"` during install and refuse with their own clear, standard error.

## Consequences

- `_cli.py` gains a `-v`/`--verbose` flag; `main()` calls `logging.basicConfig(level=logging.DEBUG, stream=sys.stderr)` once when set, before any file processing. No change to argument parsing for existing flags, no change to `check()`/`fix()` behavior, no change to the exit-code contract (`0012`).
- `tests/test_main.py` gains two subprocess-based regression tests; `tests/test_orchestrator.py` gains one in-process coverage test for the same branch.
- No other production code changed — chapters 26 and 29 were already fully satisfied by this codebase's existing design.
