# Investigation report: behavioral contract audit, configuration & environment/project-root discovery (ch. 17, 28)

Detailed findings behind `docs/adr/0019-behavioral-contract-audit-configuration-and-discovery.md`. `docs/behavioral_contract.md` chapters **17** (Configuration) and **28** (Configuration and Environment Discovery) were checked line-by-line against `main()`'s own CLI argument handling (`ast_checks/_cli.py`), `load_checks()`/`CheckOrchestrator._generate_cache_key()` (`ast_checks/_orchestrator.py`), and `CacheManager` (`_cache.py`), following on from `0011`–`0017`'s prior audits, per issue #35. Both chapters are written for a tool with a config-file format and a project-root-discovery step; this project has neither — grepped and confirmed empty (no `tomllib`, `pyproject`, `.toml`, `os.environ`, `os.getenv`, or any upward directory walk anywhere in `src/`, echoing `0014`'s identical finding for chapter 11). Every check's configuration comes exclusively from CLI arguments (`--select`/`--ignore`/`--fix`/`--exclude`, per `0008`'s CLI-parity design, plus each check's own optional `add_cli_arguments()`/`cli_kwargs_from_args()` extension point — unused by every check today), and there is no notion of a "project" beyond the literal file paths the caller passes in.

Most items were already satisfied: `--select`/`--ignore` are validated against the real check-id set before `CheckOrchestrator` is even constructed, so a destructive `--fix` run never starts against a misconfigured check set; `--select`/`--ignore` composition is deterministic and documented in `load_checks()`'s own docstring; `_generate_cache_key()` already fingerprints every enabled check's own config alongside the check-id set, so a configuration change already invalidates the cache; `--fix` is never implied by any other flag; and default-run output is already fully sorted, so the same input produces byte-identical stdout/stderr across repeated runs.

## Gaps found and fixed

- **A trailing, leading, or doubled comma in `--select`/`--ignore` (e.g. `--select=forbid-vars,`) produced a spurious `Error: Unknown checks: ` with nothing named after the colon**, instead of being tolerated the same way `--exclude`'s own comma-list parsing already tolerates it (ch. 17: "MUST report invalid configuration clearly", "MUST provide useful errors for invalid rule selectors"). Root cause: `{c.strip() for c in args.select.split(",")}` kept the empty string produced by the extra comma, which then failed the `- all_check_ids` membership check like any other unrecognized name. Reproduced concretely: `ruff-extra-rules --select=forbid-vars, file.py` printed `Error: Unknown checks: ` and exited 1, even though `forbid-vars` is a real, correctly-spelled check. Fixed by filtering blank tokens the same way `--exclude`'s own comma-list parsing already does, for both `select` and `ignore`. A `--select` value that names no real check once blanks are filtered (e.g. `--select=,,`) now falls through to the pre-existing, already-tested `Error: No checks enabled` path instead. Regression tests: `test_main_trailing_comma_does_not_report_blank_unknown_check` (parametrized over `--select`/`--ignore`), `test_main_select_only_commas_reports_no_checks_enabled` in `tests/test_orchestrator.py`.

- **`Error: Unknown checks: X` never said whether `X` came from `--select` or `--ignore`**, leaving the two indistinguishable when only one was actually misspelled (ch. 17: "MUST identify the configuration source and location when possible"). Fixed by naming the offending flag in the message (`Error: Unknown checks in --select: X` / `Error: Unknown checks in --ignore: X`), and folding the two previously near-identical `if select: ...` / `if ignore: ...` blocks into one loop over `(flag_name, check_ids)` pairs. Regression test: `test_main_unknown_check_name_returns_one` (pre-existing, updated to assert the flag name now appears in the message).

- **The cache directory's location relative to the caller's current working directory, rather than any independently-discovered project root, was never written down anywhere** (ch. 17: "MUST document defaults that materially affect analysis or fixing"; ch. 28: "MUST define how the project root is discovered"). `CacheManager.DEFAULT_CACHE_DIR` is a plain relative path, `.cache/pre_commit_hooks/`, with no code anywhere that walks up from cwd looking for `.git` or `pyproject.toml` — the "project root" is simply wherever the process's cwd happens to be. This is deliberate, matching `mypy`'s own `.mypy_cache` convention (confirmed empirically) rather than `ruff`'s (`.ruff_cache` is anchored at the project root `ruff` itself discovers, confirmed with `ruff check --show-settings`) — so this hook's own behavior is the `mypy` shape, not the `ruff` one. It is also correct in the one supported production path (`prek`/`pre-commit` both invoke every hook with cwd fixed at the repository root) and consistent for the direct-CLI dev path documented in `AGENTS.md`. It was never a behavioral bug — running the direct CLI from a different subdirectory just gets its own separate, correctly-scoped cache there, the same tradeoff `mypy` itself accepts — but it was undocumented. Fixed by adding a "Cache Location" note to README's existing `## Configuration` section stating the convention and the cwd assumption explicitly, rather than adding project-root discovery this single-maintainer local hook doesn't need. No regression test: this is a documentation-only fix for an already-correct, already-deterministic behavior.

## Items judged not applicable, with rationale

- **Ch. 17, "MUST identify the configuration source and location when possible" (config-_file_ location); "MUST NOT accidentally apply configuration from an unrelated parent directory"; "MUST define behavior when multiple configuration files are present"; "MUST define behavior when configuration files contain unsupported versions or schemas"; ch. 28, "MUST define how configuration files are discovered"; "MUST NOT silently use a configuration file from an unexpected directory"; "MUST handle missing configuration files gracefully"; "MUST handle malformed configuration files clearly"**: this project reads no configuration file of any kind — grepped and confirmed empty, the same finding `0014` already made auditing chapter 11's cache-identity requirements. Every one of these items presupposes a config-file format this tool doesn't have.
- **Ch. 17, "MUST ensure that configuration from the intended project is used"**: with no config-file discovery, there's no "intended project's config" to fail to select — the only configuration is the CLI arguments the caller passed to this exact invocation.
- **Ch. 28, "MUST define how the project root is discovered"**: this tool has no independent concept of a project root distinct from the caller's cwd (see the cache-location gap above, now documented) or the literal file paths given on the command line.
- **Ch. 28, "MUST define how environment variables affect behavior"; "MUST make environment-dependent behavior explicit"; "MUST ensure that environment variables cannot silently override explicit command-line configuration unless documented"**: grepped for `os.environ`/`os.getenv`/`environ[` across `src/pre_commit_hooks` — no check, the CLI, the cache, or the prefilter reads a single environment variable. Every text file this pipeline reads or writes already uses an explicit `encoding="utf-8"`, never a locale-dependent default. Only `git`/`git grep` subprocess resolution depends on `$PATH`, the same OS-level dependency every tool that shells out to another binary has.
- **Ch. 28, "MUST define precedence between command-line arguments, environment variables, configuration files, and defaults"**: collapses to "an explicit CLI argument overrides argparse's own default" — the only two layers that exist here.
- **Ch. 28, "MUST ensure that the same invocation produces predictable results from different working directories when the selected project is the same"; "MUST avoid accidentally depending on the caller's current working directory when the intended project root is known"**: every check's actual violation output depends only on each file's own resolved content, never on cwd — `git grep`'s own prefiltering is a performance optimization with an already-audited (`0015`) Python fallback that produces the identical candidate set. The one place cwd does matter, the cache directory's location, is now documented above rather than hidden.

## Consequences

- `_cli.py`'s `--select`/`--ignore` parsing now filters blank comma-separated tokens, matching `--exclude`'s pre-existing behavior; a value that resolves to no real check now reports `Error: No checks enabled` instead of a blank `Error: Unknown checks: `.
- Both "unknown check" error paths are now one loop over `(flag_name, check_ids)` instead of two near-duplicate `if` blocks, and the printed message now names `--select`/`--ignore` explicitly.
- README's `## Configuration` section gains a "Cache Location" note stating that `.cache/pre_commit_hooks/` is resolved relative to the process's cwd, not an independently discovered project root, and that `prek`/`pre-commit` already guarantee the one location that matters (the repository root) for the hook's own supported entry point.
