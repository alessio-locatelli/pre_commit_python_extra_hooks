# Investigation report: behavioral contract audit, cross-platform behavior (ch. 14)

Detailed findings behind `docs/adr/0020-behavioral-contract-audit-cross-platform-behavior.md`. `docs/behavioral_contract.md` chapter **14** (Cross-Platform Behavior) was checked line-by-line against every module in `src/pre_commit_hooks/`, following on from `0011`–`0019`'s prior audits and `0015`'s chapter-13 path-handling audit it's sequenced after, per issue #42. Grepped the whole source tree for platform-sensitive stdlib usage (`fcntl`, `resource`, `termios`, `tty`, `pwd`, `grp`, `msvcrt`, `winreg`, `curses`, `os.fork`, `sys.platform`, `os.name`) and for any `subprocess.run()`/`Popen()` call passing a POSIX-only keyword — the only hit anywhere was `_cache.py`'s own `import fcntl`, described below.

Most items were already satisfied. Every path throughout the pipeline goes through `pathlib.Path`, never a hand-built separator or a platform-specific normalization rule (`0015`'s chapter-13 audit already verified this in depth). `atomic_write_text()` and `_cache.py`'s own `_write_cache()` both use `tempfile.mkstemp()`/`Path.replace()`, explicitly documented as "atomic on POSIX" — the one platform this project targets — using only cross-platform-safe stdlib calls with no POSIX-only keyword arguments. `os.access()`'s writability check and every `OSError`-handling path around permission-denied files already handle POSIX permission bits the only way this platform has them. Every text read/write already passes an explicit `encoding=` and `newline=""` (`0004`), so a file's line endings are preserved byte-for-byte regardless of platform. `__main__.py`'s SIGTERM handler installation (`0016`) already wraps `signal.signal()` in `try: ... except ValueError, OSError:` and degrades to running without it rather than crashing. No check performs permission-bit or symlink logic that assumes a Unix-only filesystem feature beyond what `0015`'s chapter-13 audit already covered, and nothing anywhere references a Windows-specific filesystem concept.

## Gap found and fixed

- **`_cache.py`'s unconditional `import fcntl` at module load time made the whole package fail to import on any platform without it (Windows has no `fcntl` module at all), before a single line of this hook's own error handling could ever run** (ch. 14: "MUST NOT hard-crash merely because an optional platform feature is unavailable"; "MUST either implement platform-specific behavior correctly or gracefully degrade with a clear warning"; "MUST handle platform-specific file locking behavior correctly"). `fcntl.flock()` is used in exactly one place, `CacheManager._locked()`'s advisory cross-process lock guarding the read-modify-write of a per-file cache blob against a deterministic, shared `*.tmp` path (hardened for timeout by `0013`) — under `prek`'s parallel execution, two processes racing to update the same cache entry unlocked can interleave writes to that shared temp file and leave a corrupted or wrong-but-still-parses-as-valid-JSON blob behind, not just lose one process's update cleanly. `AGENTS.md` previously stated this project assumes Linux/WSL only and forbade any platform branching at all, which is what let this ship unnoticed: nothing exercised the on-a-platform-without-`fcntl` path because the project's own stated design ruled out testing it. Fixed by wrapping the import in `try: import fcntl / except ImportError: fcntl = None`, and computing `self._locking_unavailable = fcntl is None` once in `CacheManager.__init__` (mirroring `_cache_dir_unavailable`'s existing "check once at construction, not per file" shape), logging one clear `logger.warning(...)` naming `os.name` when it's set.

  An earlier draft of this fix made only `_locked()` degrade to an unlocked no-op, keeping the cache itself enabled — this reintroduces exactly the unsynchronized read-modify-write race `_locked()`'s own docstring says it exists to prevent, just on a platform where the lock happens to be unavailable rather than absent by an oversight. The cache is disabled entirely instead: `get_cached_result()`/`set_cached_result()` now short-circuit on `self._cache_dir_unavailable or self._locking_unavailable`, the same way they already do for an unavailable cache directory. Every check still runs correctly without a cache, just uncached, on a platform without `fcntl` — the same degraded-but-correct behavior `_cache_dir_unavailable` already provides for a read-only cache directory.

  Regression tests (`tests/test_cache.py`): `test_construction_warns_and_continues_when_fcntl_unavailable` (asserts `_locking_unavailable` is set and a "fcntl" warning is logged, monkeypatching the already-imported module-level name rather than trying to fake away a stdlib module at interpreter-import time — the same pattern `test_main.py`'s `test_install_sigterm_handler_degrades_when_signal_signal_unavailable` uses) and `test_cache_disabled_entirely_without_fcntl` (proving `get_cached_result`/`set_cached_result` behave as a permanent, silent-to-the-caller miss — no `.json` cache file and no `.lock` file ever written — with `fcntl` unavailable). Both fail against the pre-fix code path: the first by crashing at import, the second by asserting a `None` result where the earlier unlocked-cache draft would have returned the written value.

## Items judged not applicable, with rationale

- **Ch. 14, "MUST support all officially supported operating systems"; "MUST NOT claim cross-platform support if core functionality is untested on a supported platform"**: the only "officially supported" platform is Linux/WSL (`AGENTS.md`, `pyproject.toml`'s `Operating System :: POSIX :: Linux` classifier). Nothing in this project claims Windows or macOS support; the fix above changes an unsupported platform's failure mode from "crashes on import" to "runs with a warning," it does not add or claim new platform support.
- **Ch. 14, "MUST handle Windows path semantics correctly when Windows is supported"; "MUST not assume that Windows-specific filesystem behavior exists on POSIX"**: Windows isn't a supported platform (`AGENTS.md`), so there's no Windows path semantics to implement.
- **Ch. 14, "MUST provide a reasonable fallback when optional acceleration is unavailable"; "MUST ensure that unsupported optimizations do not become correctness requirements"**: satisfied by the `fcntl` fix above and, separately, already satisfied by `0015`'s chapter-13 audit for `_prefilter.py`'s `git grep` fast path, which already falls back to a pure-Python scan producing an identical candidate set.
- **Ch. 14, "MUST test supported platforms in CI or through an equivalent reliable validation process"**: no hosted CI exists for this single-maintainer local repo (`AGENTS.md`) — the full `Development` command sequence, run locally on the one supported platform before every commit, is the "equivalent reliable validation process" the item itself allows for.

## Consequences

- `_cache.py`'s `fcntl` import is now conditional; `CacheManager` gains a `_locking_unavailable` attribute (checked once in `__init__`, mirroring `_cache_dir_unavailable`) and logs one warning naming `os.name` when set. `get_cached_result()`/`set_cached_result()` now treat `_cache_dir_unavailable` and `_locking_unavailable` identically — either one disables the cache for the rest of the run, degrading to uncached-but-correct execution rather than proceeding unlocked.
- `AGENTS.md`'s platform-assumption line now states the degrade-with-a-warning requirement explicitly alongside the pre-existing "don't add Windows/macOS-specific code paths" instruction, pointing at this ADR.
- No behavior changes on the one supported platform (Linux/WSL, where `fcntl` always imports successfully): `self._locking_unavailable` is `False` there, so caching and `_locked()`'s own locking take the exact same path as before.
