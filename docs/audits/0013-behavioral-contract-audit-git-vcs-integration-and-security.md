# Investigation report: behavioral contract audit, Git/VCS integration & security (ch. 22, 23)

Detailed findings behind `docs/adr/0024-behavioral-contract-audit-git-vcs-integration-and-security.md`. `docs/behavioral_contract.md` chapters **22** (Git and VCS Integration) and **23** (Security) were checked line-by-line against every subprocess call to `git` (`_prefilter.py`'s `git_grep_filter()`, `ast_checks/_discovery.py`'s `_list_python_files_in_dir()`), every temporary/cache-file write (`_cache.py`'s `_write_cache()`/`_ensure_cache_dir()`/`_locked()`, `ast_checks/_base.py`'s `atomic_write_text()`), and the rest of the pipeline for source-execution, config-execution, and injection risk, following on from `0011`–`0022`'s prior audits, per issue #43.

Most items were already satisfied. No code anywhere `eval`s, `exec`s, or dynamically imports analyzed source or project configuration. There is no config-file parsing anywhere in the pipeline (`0014`), so "treat repository configuration as potentially untrusted" has nothing to apply to. Every subprocess call builds its argv as a literal list with a hardcoded git subcommand and an explicit `--` pathspec separator — grepped for `shell=True`/`os.system`/`os.popen`, none found — re-verified empirically with a directory name containing `;&&|$(touch pwned)` passed to `git -C <dir> ls-files`. `batch_filter_files()` always calls `git_grep_filter()` with `fixed_string=True` in every real call path, so a hostile repo's `grep.patternType` config can never reinterpret a prefilter string as a regex. `atomic_write_text()` already creates its temp file via `tempfile.mkstemp()` (unpredictable name, mode 0600, `O_EXCL`) rather than a fixed sibling. VCS integration is entirely read-only — `git grep`/`git ls-files` are the only two git subcommands invoked anywhere, neither a write operation.

## Gaps found and fixed

- **`git_grep_filter()` silently dropped an explicitly-named file that git's index doesn't know about — untracked (never `git add`ed) or matched by `.gitignore` — from every check's candidate list, with zero trace** (ch. 22: "MUST define whether ignored files are analyzed"; ch. 12: "MUST process only the requested scope"; ch. 34: "MUST NOT silently ignore failures"). Plain `git grep <pattern> -- <path>` only searches tracked files, even when a path is named explicitly: reproduced concretely inside a real git repo — a brand-new, readable, never-staged `.py` file containing a real prefilter-matching pattern makes `git grep` exit 1 with empty stdout and empty stderr, the exact signal `git_grep_filter()`'s existing "0 or 1 with no stderr means trust the result" fast path already treats as authoritative proof of "no match" (the same failure shape `0015`'s "file vanished since discovery" fix already hardened against, just from a different cause: not deleted, never indexed in the first place). Since 5 of this project's 6 checks declare a prefilter pattern, this was the default no-flags code path for nearly every file passed directly on the CLI before it had ever been `git add`ed — exactly the moment a developer is most likely to want to lint it. Pre-commit/prek's own normal invocation is unaffected: a file only reaches a hook after `git add` stages it, which already makes it visible to a plain `git grep`. Fixed by adding `--untracked --no-exclude-standard` to the `git grep` invocation, so it searches the working-tree content of every path named in its own `--` pathspec list regardless of index/ignore status. Deliberately scoped to `git_grep_filter()` only, not `_discovery.py`'s `_list_python_files_in_dir()`: that function's own `.gitignore`-aware `git ls-files` is a directory scan ("what's currently under here"), where excluding untracked scratch files and build artifacts living under the given directory is the documented, tested, intentional behavior (`0015`) — the gap here is specific to a path the caller named explicitly, which chapter 12 already treats as unconditionally in scope. Regression tests: `test_git_grep_filter_includes_untracked_file`, `test_git_grep_filter_includes_gitignored_file` (`tests/test_prefilter.py`, real `git` subprocess, no mocking); both fail against the pre-fix code.

- **`_cache.py`'s `_write_cache()` wrote through a fixed `<hash>.tmp` sibling (`cache_file.with_suffix(".tmp")`) via a plain `open(..., "w")`, instead of `atomic_write_text()`'s own already-hardened `tempfile.mkstemp()` pattern** (ch. 23: "MUST ensure that temporary files are created safely", "MUST avoid predictable temporary-file names where attackers could exploit them"). The temp path is fully predictable from the cache file's own name (a SHA-1 of the resolved source-file path), and a plain `open("w")` follows a symlink rather than refusing to. Reproduced concretely: pre-planting a symlink at that exact predictable path, pointing at an unrelated file, made `_write_cache()` overwrite the symlink's target with cache JSON content on its very next write — a classic TOCTOU symlink attack, exploitable by anyone able to write to `.cache/pre_commit_hooks/` (the same trust boundary `0015`'s NFS item already accepted for this single-maintainer local-cache design, but here fixable for free rather than requiring new capability). Fixed by switching to `tempfile.mkstemp(dir=cache_file.parent, ...)`, mirroring `atomic_write_text()`'s own established pattern exactly — the file is created fresh and exclusively, so a pre-planted symlink at the old fixed name is simply never opened. Regression test: `test_write_cache_does_not_follow_a_symlink_planted_at_the_old_predictable_temp_name` (`tests/test_cache.py`), which pre-plants a symlink at the old fixed path pointing at a "victim" file and asserts it's left untouched; fails against the pre-fix code. `test_write_cache_cleans_up_temp_file_on_write_error` was updated to glob for any leftover `*.tmp` file rather than asserting the absence of the specific old fixed name.

## Items judged not applicable, with rationale

- **Ch. 23, `_ensure_cache_dir()`'s `CACHEDIR.TAG` write follows the same symlink-following pattern the `_write_cache()` gap above had**: unlike that gap, this is a one-time write per fresh cache directory of a fixed, public, non-sensitive banner (the [bford.info/cachedir](https://bford.info/cachedir/) marker text) — not cache content an attacker could use to make the linter report a false clean result. Not hardened: disproportionate for a decorative, low-value, single-write marker file with no correctness or confidentiality stake.
- **Ch. 23, `_locked()`'s lock file (`cache_file.with_suffix(".lock")`) is also a predictable, symlink-followable path**: unlike a temp file, a lock file's name must be predictable and shared — that's the only way two independent processes agree on which file to serialize against. A pre-planted symlink there would make the process `flock()` an attacker-chosen file, but nothing is ever written through it (only `open(..., "a")` for the fd and `fcntl.flock()` against it), so the blast radius is limited to spurious lock contention on a file the attacker chose — never data corruption. Not hardened, for the same "requires the same write access the attacker would already have" reason as the item above.
- **Ch. 22, "MUST handle files modified after the linter discovered them" / "MUST provide a safe behavior when a file changed concurrently"**: already audited and judged not applicable by `0011` — nothing git-specific changes that analysis, since these checks never read from or write to the git index at all.
- **Ch. 22, "MUST handle repositories with unusual paths and worktree layouts correctly"**: neither `git_grep_filter()` nor `_list_python_files_in_dir()` ever parses `.git` itself or hardcodes an assumption about repository layout — locating the actual repository is entirely git's own responsibility, the same as `mypy`/`ruff` themselves rely on.
- **Ch. 23, "MUST NOT assume that a repository is trusted merely because it is local" / "MUST document any behavior that executes external programs or project-defined code"**: no trust decision is made anywhere that a hostile local repository could hijack (no config parsing, no code execution, `grep.patternType` pinned via `--fixed-strings`), and both `git` subcommands this project invokes are already documented in `docs/adding-a-check.md` and `CONTEXT.md`.
- **Ch. 23, "MUST ensure that diagnostic output cannot be interpreted as executable shell input without appropriate escaping"**: diagnostics are plain `file:line:col: CODE: message` text printed to stderr; this project never feeds its own diagnostic output back into a shell anywhere.

## Consequences

- `git_grep_filter()`'s `git grep` invocation now always includes `--untracked --no-exclude-standard`; on a repository with a very large number of untracked/ignored files this makes git's own working-tree scan marginally more expensive, but the command still only ever searches the exact pathspec list the caller passed in.
- `_cache.py` gains a `tempfile` import; `_write_cache()`'s temp files are now named `.{cache_file.name}.<random>.tmp` instead of `<hash>.tmp`, matching `atomic_write_text()`'s own naming convention. No observable behavior change for a well-behaved concurrent run — this only changes what an attacker pre-planting a file at the old fixed name can achieve, not normal operation.
