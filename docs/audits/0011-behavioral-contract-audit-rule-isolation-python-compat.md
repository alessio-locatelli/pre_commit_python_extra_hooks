# Investigation report: behavioral contract audit, rule isolation & Python-version compatibility (ch. 18, 19)

Detailed findings behind `docs/adr/0021-behavioral-contract-audit-rule-isolation-python-compat.md`. `docs/behavioral_contract.md` chapters **18** (Rule and Plugin Isolation) and **19** (Python Version and Language Compatibility) were checked line-by-line against `ASTCheck`/`atomic_write_text()` (`ast_checks/_base.py`), `CheckOrchestrator`/`load_checks()`/`_generate_cache_key()` (`ast_checks/_orchestrator.py`), `ALL_CHECKS` (`ast_checks/__init__.py`), `_cli.py`'s CLI wiring, and every check's own `fix()`, following on from `0011`–`0020`'s prior audits, per issue #36.

Both chapters are written for a linter with a dynamic or third-party plugin system and a configurable analysis target version — this project has neither. There is no plugin mechanism at all: `ALL_CHECKS` is a fixed, hand-authored `list[type[ASTCheck]]` of in-repo classes, with a comment stating exactly that ("no plugin mechanism for third-party checks, so a static list is all that's needed"). And there is no separate "target Python version" concept independent of the interpreter actually running the hook: `AGENTS.md` states this project "support[s] only the current stable Python version (currently `>=3.14`)", and grepping confirms no `ast.parse`/`compile` call anywhere in `src/` passes a `feature_version` or otherwise selects a grammar different from the running interpreter's own. Every syntax-related item in chapter 19 collapses to "whatever the running interpreter's own `ast`/`compile` accepts."

## Already satisfied, confirmed with citations

- **Ch. 18, "MUST define the interface between rules and the core engine"**: the `ASTCheck` `Protocol` (`_base.py:41-119`) is that interface, and `docs/adding-a-check.md` documents it a second time for a human author, including the constraint that "`check()` must only ever look at the single file it's given — no check reads another file, imports, or any other cross-file state."
- **Ch. 18, "MUST ensure that a rule cannot accidentally modify source outside its declared scope"**: every check's only write path is `atomic_write_text(filepath, ...)`, targeting exactly the one `filepath` the check itself was given — confirmed none performs any other file, network, or environment I/O.
- **Ch. 18, "MUST validate rule-provided edits before applying them"; "MUST apply global validation to all fixes regardless of their source"**: `atomic_write_text()` is the one function every check's `fix()` writes through, and it always `compile()`s the proposed content before any file I/O.
- **Ch. 18, "MUST handle malformed rule output safely"; "MUST isolate failures ... where practical"**: `_check_file()`'s and `_apply_fixes()`'s per-check loops each wrap a check's `check()`/`fix()` call in its own `try`/`except Exception`, recording `(filepath, check_id)` in `rule_failures` — audited in depth by `0012`.
- **Ch. 18, "MUST identify the rule responsible for a failed diagnostic or fix"**: `rule_failures` records the failing `check_id` alongside the file.
- **Ch. 18, "MUST invalidate caches when rule implementation behavior changes"**: `_generate_cache_key()` hashes every enabled check's `check_id`, its constructor-argument fingerprint, a SHA-1 of this whole package's source tree, and the running interpreter's `major.minor` — any change to a check's code invalidates every cached result (`0005`, `0014`).
- **Ch. 18, "MUST ensure that disabled rules do not produce diagnostics or fixes"; "MUST ensure that rule selection is deterministic"**: `load_checks()` only instantiates checks passing the `select`/`ignore` filter, iterating `ALL_CHECKS`'s fixed order.
- **Ch. 19, "MUST define the supported Python language versions"**: `pyproject.toml`'s `requires-python = ">=3.14"` and `AGENTS.md`.
- **Ch. 19, "MUST analyze syntax according to the configured target version"; "MUST NOT silently interpret newer syntax as older syntax incorrectly"; "MUST reject or clearly report unsupported syntax"; "MUST ensure that auto-fixes are valid for the configured target version"**: there's exactly one "configured target version" — the running interpreter — so `ast.parse()` always parses using its own grammar, and `atomic_write_text()`'s `compile()` call already validates every fix against it. A `SyntaxError` is caught by `_check_file()` and reported via `unprocessable_files` (`0011`).
- **Ch. 19, "MUST include the target version in relevant cache identity"**: `_generate_cache_key()` includes `f"{sys.version_info.major}.{sys.version_info.minor}"` (`0014`).
- **This project's own use of PEP 758 (`except A, B:`, no parentheses) syntax is itself a live, continuously-exercised proof of the one-target-version design working correctly**: `.pre-commit-config.yaml`'s own `ruff-extra-rules` hook runs this project's checks against `src/` and `tests/` on every commit, meaning several PEP 758 clauses and a PEP 695 `type` alias statement in this project's own source are parsed and walked by its own checks on every commit without incident.

## Not applicable, with rationale

- **Ch. 18, "MUST isolate failures in third-party or dynamically loaded rules where practical"; "MUST define compatibility requirements for plugins"**: isolation and in-house compatibility requirements do exist (above), but the literal "third-party"/"dynamically loaded" framing doesn't apply — `ALL_CHECKS` is a compile-time-fixed list of classes defined in this same package, never discovered via an entry-point or dynamic import.
- **Ch. 19, "MUST test rules across all supported language versions where behavior differs"**: there is exactly one supported language version, so there is no cross-version behavior difference to test.

## Regression test added

No behavior in either chapter was actually wrong — the audit didn't reproduce a single case of two checks sharing a `check_id`/`error_code`, a fix bypassing `atomic_write_text()`, or a version mismatch. One item was true today but unverified: `docs/adding-a-check.md` already documents "next unused number" as the convention for picking a new check's `check_id`/`error_code`, but nothing enforced it — a future check added with a copy-pasted or mistyped `check_id` would silently run alongside the check it collided with. `test_all_checks_have_unique_check_ids_and_error_codes` (`tests/test_orchestrator.py`) locks the current, already-correct state in going forward.

## Consequences

- `tests/test_orchestrator.py` gains `test_all_checks_have_unique_check_ids_and_error_codes`, asserting `ALL_CHECKS` never collides on `check_id` or `error_code`. No production code changed.
