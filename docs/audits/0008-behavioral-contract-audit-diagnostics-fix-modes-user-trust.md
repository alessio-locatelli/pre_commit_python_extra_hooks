# Investigation report: behavioral contract audit, diagnostics, auto-fix modes & user trust (ch. 7, 8, 34)

Detailed findings behind `docs/adr/0017-behavioral-contract-audit-diagnostics-fix-modes-user-trust.md`. `docs/behavioral_contract.md` chapters **7** (Diagnostics and User Feedback), **8** (Auto-Fix Modes), and **34** (User Trust) were checked line-by-line against `CheckOrchestrator` and `main()`'s own reporting (`ast_checks/__init__.py`), the four checks with their own file-write logic, and every `logger.exception()` call site across `src/`, following on from `0011`–`0016`'s prior audits, per issue #41.

Most items were already satisfied: check mode never writes, `[FIXABLE]`/`[FIXED]`/`[FIX REJECTED]`/`[FIX ERRORED]` already gave a partially-fixable file's diagnostics a clear, per-violation outcome, `main()` never claims every violation was fixed, and every diagnostic already carries a file, line, rule identifier, and human-readable message.

## Gaps found and fixed

- **Every diagnostic's own column was silently dropped, and — once added — two of the checks reporting one were reporting the wrong number.** `main()`'s printed line only ever interpolated `v.line`, never `v.col` (ch. 7: "MUST report line and column information accurately when available"). Fixed by adding the column to the printed line — but simply printing `v.col` as-is was itself inaccurate for two of the checks that populate it: `forbid_vars` and `redundant_assignment` both stored `ast.col_offset` directly, a UTF-8 _byte_ offset, not a character offset (`_base.py`'s own pre-existing `byte_col_to_char_col()` already documents the distinction, and was already used by both checks' own fix logic for exactly this reason) — a line with non-ASCII text before the violation reported a column too far right. `misplaced_comment`'s own `tokenize`-derived column was already a character offset, needing no change. Fixed by converting `forbid_vars`' and `redundant_assignment`'s own diagnostic-facing column with `byte_col_to_char_col()` at the one place each actually constructs its `Violation` (their own fix logic already re-derives its edit positions fresh from the tree rather than reading this stored value back, so this only affects what's reported, never what's edited). `main()` also now reports the conventional 1-based column (`v.col + 1`) instead of the AST's own native 0-based one, matching `ruff check`'s own convention (`0008`). Regression tests: `test_main_reports_column_alongside_line` (updated), `test_check_reports_character_offset_not_byte_offset_before_multibyte_text` in `tests/test_forbid_vars.py` and `tests/redundant_assignment/test_check.py`.

- **`ASTCheck.fix()`'s own documented `True`/`False`-never-raises contract for a caught `OSError` was never actually inspected by its only caller.** `_base.py`'s own docstring already told every check to catch an `atomic_write_text()` `OSError` (disk full, permission denied, missing parent directory) itself and return `False` rather than let it propagate — and `excessive_blank_lines`, `misplaced_comment`, `forbid_vars`, and `redundant_assignment` all already did exactly that. But `CheckOrchestrator._apply_fixes` never read the return value at all: after a caught `OSError`, the violation was left in its original, unmarked state and reported as ordinary `[FIXABLE]` with "Run with --fix to inline automatically" — actively misleading, since `--fix` had already been attempted and had already failed (ch. 7: "MUST clearly report when a fix was not applied"; ch. 8: "MUST define how a partially fixable file is handled"; ch. 34: "MUST NOT silently ignore failures"). This is the same class of gap `0012` already fixed for an unexpected exception during `fix()`, but never covered this separate, already-documented "caught its own I/O error" outcome. Fixed by completing the three-way contract `_base.py`'s docstring already implied: a new `mark_fix_failed()`/`is_fix_failed()` pair, wired into each of the four checks' own `except OSError` branch, and a new `[FIX FAILED]` tag in `main()`'s report loop with its own hint ("check file permissions and available disk space, then run with --fix again") — deliberately not reusing `[FIX ERRORED]`'s "this is a bug, please report it" wording, since a permission-denied or disk-full condition isn't a bug in the check's own logic. Regression tests: `test_mark_fix_failed`/`test_is_fix_failed_false_when_only_marked_errored` in `tests/test_base.py`; `test_fix_write_failure_returns_false` (extended) in each of `tests/test_excessive_blank_lines.py`, `tests/test_misplaced_comment.py`, `tests/test_forbid_vars.py`, `tests/redundant_assignment/test_autofix.py`; `test_main_fix_flag_reports_failed_fix` in `tests/test_orchestrator.py`.

  Two further, structurally identical gaps turned up in `_apply_fixes` itself while fixing the above: its mid-loop re-read (done again before each fixable check's own recompute, in case an earlier check in the same loop already modified the file) and its own broad outer `except Exception` (covering the re-parse and the fresh-violations recompute) both `continue`d or logged with zero user-facing signal — no `rule_failures` entry, no violation marking. A check whose re-read or recompute failed this way kept its stale pre-fix snapshot and was reported as ordinary `[FIXABLE]`, exactly as misleading as the `OSError` gap above. Fixed the same way `0012` already fixed the equivalent gap for `fix()` itself raising: both paths now append to `self.rule_failures` and call `mark_fix_errored()` on the check's own still-fixable stale entries. Regression tests: `test_main_reports_rule_failure_when_reread_fails_mid_fix_loop`, `test_main_reports_rule_failure_when_recompute_raises_mid_fix_loop` in `tests/test_orchestrator.py`.

- **A violation left open after its own check's `fix()` ran never got its own stale, pre-fix line number refreshed again, even when a later check's fix in the same run shifted every line after it.** `_apply_fixes`'s existing recompute-before-fix already handled the fixable-vs-fixable case (`0011`), but only once, immediately before that check's own `fix()` call — never again afterward. Two distinct shapes: a check that's never fixable at all (`redundant-super-init`) never enters the per-check loop, so its very first, pre-any-fix `check()` result is carried through to the final report untouched; and a check that did run its own `fix()` this call but left one of its own violations open (e.g. `validate-function-name`'s `should_autofix` guard refusing to touch a method while renaming an unrelated free function in the same call) has that specific violation frozen at whatever position it had when that check's own recompute ran, not the file's true final state (ch. 7: "MUST report line and column information accurately when available"; ch. 34: "MUST NOT silently use stale analysis results when they can affect correctness"). Fixed with a final reconciliation pass, `CheckOrchestrator._refresh_stale_positions()`, run once after the per-check loop finishes, but only when the loop actually resolved at least one violation this call, so a run where every fix attempt was rejected or failed outright doesn't pay for an extra read+parse+recheck for nothing. The pass considers each check_id's whole current group of entries together: if any of them already carries a `fix_rejected`/`fix_errored`/`fix_failed` mark, that check_id is skipped entirely this pass — a fresh `check()` call would rediscover that same still-present violation too, and there's no reliable way to tell that rediscovery apart from a genuinely different, unrelated violation that merely happens to share message text with it, for lack of a stable per-violation identity that survives a `check()` call in this codebase. An earlier version of this fix tried to filter the fresh result by message instead, which was caught silently dropping exactly that kind of unrelated violation from the report entirely — worse than leaving a position stale (ch. 34: "MUST prefer a visible failure over a silent incorrect result"). Only when a check_id has no rejected/errored/failed entries at all does the pass drop its still-open entries and replace them with that check's fresh `check()` result against the file's true final state. Regression tests: `test_apply_fixes_refreshes_non_fixable_checks_position_too`, `test_apply_fixes_refreshes_a_participating_checks_own_left_open_violation`, `test_refresh_stale_positions_never_drops_an_unrelated_open_violation_sharing_a_check_id_with_a_terminal_one` in `tests/test_orchestrator.py`.

- **Several internal `logger.exception()` calls leaked a full raw Python traceback onto the ordinary user's stderr by default, duplicating and cluttering the clean diagnostic line the hook already prints for the same failure.** Nothing in this codebase configures logging (no `basicConfig()`, no handler, no level anywhere), so every `logger.warning()`/`logger.exception()` call falls through to Python's own `logging.lastResort` handler, which prints any WARNING-or-above record straight to `sys.stderr` by default — confirmed by running the hook as a real subprocess against a permission-denied file: three separate tracebacks, one from `_prefilter.py`'s own git-grep-unavailable fallback and two duplicates from `_read_source`, all printed before the tool's own single, documented `file: error: could not be read or parsed; file skipped` line (ch. 7: "MUST NOT emit uncontrolled human-oriented text into a machine-readable output stream"; ch. 34: "MUST make errors actionable"). Fixed by downgrading every `logger.exception()` call whose failure is also independently and cleanly reported elsewhere to `logger.debug(..., exc_info=True)` — preserving the traceback for a developer who explicitly raises the log level, while no longer polluting the ordinary user's stderr by default. `ruff`'s own `BLE001` rule specifically recognizes a `logger.exception(...)` call as evidence an `except Exception:` isn't swallowing its error, but doesn't recognize `logger.debug(..., exc_info=True)` the same way; each of the five affected `except Exception:` blocks got a one-line `# noqa: BLE001` explaining why. `load_checks()`'s own two `logger.exception()` calls (a check's `__init__` raising) were deliberately left unchanged — `0012` already examined this exact path and decided not to extend the exit code for it. Regression tests: `caplog`-based level assertions added throughout the existing exception-path tests across the affected modules; `test_real_invocation_does_not_leak_a_traceback_onto_stderr` in `tests/test_main.py` is the one true end-to-end proof, since it's the only test in the suite that spawns a real subprocess rather than calling `main()`/`run()` in-process.

## Items judged not applicable, with rationale

- **Ch. 7, "MUST provide a machine-readable output mode suitable for CI and editor integrations"; ch. 34, "MUST ensure that editor integrations can reliably determine the current state of diagnostics"**: this hook's one output format (`file:line:col: code: [tag] message`, one diagnostic per stderr line) is already the traditional compiler/grep-style location format most CI "problem matcher" and editor "quickfix" integrations already parse. `0008` already deliberately deferred every `ruff check` flag beyond `--select`/`--ignore`/`--fix`/`--exclude`/`--list-checks`, explicitly naming a structured output mode as the kind of flag out of scope.
- **Ch. 7, "MUST provide an appropriate quiet mode where supported"**: not promised anywhere in this hook's own CLI surface, so there's no quiet mode to keep honest.
- **Ch. 8, "MUST make unsafe or unavailable fixes distinguishable from safe fixes"**: this hook has no `--unsafe-fixes`-style concept at all — every fix this tool ever proposes is unconditionally the safe kind by construction (`0008`'s own consequences already document every check gating its own fix safety internally).
- **Ch. 8, "MUST make the relationship between fix mode and formatting mode explicit when both exist"**: already explicit at the project level — `CONTEXT.md`'s own opening line and `AGENTS.md`'s own title both state this hook "runs alongside ruff, not instead of it," and `ruff-format` is a wholly separate hook.
- **Ch. 8, "MUST NOT allow independent tools or passes to continually undo each other's changes without detecting the conflict"**: a single pre-commit hook has no visibility into what a different, externally-invoked hook does to the same file — pre-commit/prek's own convention of re-staging and re-running every hook until the working tree stops changing is the actual safety net here, which is what makes this hook's own idempotence (spot-checked by `0011`) converge rather than loop.
- **`validate_function_name`'s own internal `should_autofix()`/`apply_fix()` `except (OSError, SyntaxError, UnicodeDecodeError, LookupError)` branches were not wired into `mark_fix_failed()`.** Unlike the four checks fixed above, these two functions mix a genuine I/O failure with several purely structural, by-design declines behind the same plain `False` return — conflating them would tag a legitimate, unrelated safety-guard decision with a "check permissions and disk space" hint that doesn't apply. These two functions already log via `logger.warning()`, not `logger.exception()`, so they were never part of the traceback-leak problem this audit's logging fix addresses. Disentangling the two would need a more invasive refactor of this module's own boolean-only API, disproportionate to what this audit's scope requires; left as a deliberately deferred follow-up.

## Consequences

- `Violation`'s reported diagnostics now include the column (`file:line:col: code: message`), 1-based and always a character offset, matching the location format most CI/editor tooling (and `ruff check` itself) already expects.
- `_base.py` gains a third fix-outcome marker, `mark_fix_failed()`/`is_fix_failed()`, alongside the existing `mark_fixed()`/`mark_fix_rejected()`/`mark_fix_errored()` trio; `main()`'s report loop gains a corresponding `[FIX FAILED]` tag.
- `CheckOrchestrator._apply_fixes` gains a `file_changed` tracking flag and calls a new private method, `_refresh_stale_positions()`, once per fixed file when at least one violation was actually resolved this call.
- Several `logger.exception()` calls throughout `src/` are now `logger.debug(..., exc_info=True)` — still fully available to a developer who raises the log level explicitly, but no longer visible in a plain, unconfigured invocation's stderr by default.
