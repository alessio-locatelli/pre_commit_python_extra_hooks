# Investigation report: behavioral contract audit, process, signals, cancellation & parallelism (ch. 15, 16)

Detailed findings behind `docs/adr/0016-behavioral-contract-audit-process-signals-parallelism.md`. `docs/behavioral_contract.md` chapters **15** (Process, Signals, and Cancellation) and **16** (Parallelism and Concurrency) were checked line-by-line against `__main__.py`'s process entry point, `atomic_write_text()` (`ast_checks/_base.py`), `_cache.py`'s `_write_cache()`/`_locked()`, and `_prefilter.py`'s/`ast_checks/__init__.py`'s `subprocess.run()` calls, following on from `0011`–`0015`'s prior audits, per issue #34.

This tool runs single-process and serial within one invocation — no `threading`, `multiprocessing`, or `concurrent.futures` anywhere in `src/` (grepped, confirmed empty) — so nearly every chapter 16 bullet describes a scenario (worker crashes, competing modifications between two in-process workers, falling back from parallel to serial) that cannot occur by construction, not one this audit chose not to address. The one real multi-process scenario this codebase does have — multiple separate `prek`/pre-commit hook invocations racing on the same shared `.cache/pre_commit_hooks/` cache — was already fully hardened by `0013` (`_locked()`'s timeout-bounded `flock`) and is not revisited here. Chapter 15 had two genuine gaps, both about signal-based cancellation before this audit reaching only SIGINT (via Python's own built-in `KeyboardInterrupt` translation) and doing so ungracefully (a raw traceback, no defined exit code).

## Gaps found and fixed

- **SIGTERM had no handler at all, so the OS's default action (immediate termination) applied — skipping every `try`/`finally` cleanup this codebase already relies on** (ch. 15: "MUST handle user cancellation gracefully where the operating system provides cancellation mechanisms", "MUST clean up temporary files where possible", "MUST handle termination during file replacement safely", "MUST ensure that child processes do not remain unexpectedly orphaned when the tool is terminated"). Python installs a default handler only for SIGINT (translating it to a catchable `KeyboardInterrupt`); SIGTERM — the signal `prek`/pre-commit's own timeout handling, a CI job cancellation, or a plain `kill <pid>` would send — had no handler, so the process died mid-syscall with no chance to run `atomic_write_text()`'s or `CacheManager._write_cache()`'s existing `finally: temp_path.unlink()` cleanup, and no chance for `subprocess.run()`'s own built-in child-killing cleanup to fire either. Fixed by installing a SIGTERM handler (`ast_checks/__main__.py`'s new `_install_sigterm_handler()`) that raises `KeyboardInterrupt` — deliberately reusing Python's own SIGINT translation path rather than inventing a second cancellation mechanism, so every `try`/`finally` this codebase already has now runs for SIGTERM exactly as it already did for SIGINT. Regression test: `test_real_sigterm_mid_run_stops_gracefully_without_leftover_temp_files` (`tests/test_main.py`) — a real `os.kill(os.getpid(), signal.SIGTERM)`, fired mid-batch via a monkeypatched `CheckOrchestrator._check_file` counting real calls (not a wall-clock sleep, so the test is deterministic), asserting the run stopped before every file was processed, left no `*.tmp` files behind, and left every file on disk as either its original or its fully-fixed content — never a partial write.

- **An uncaught `KeyboardInterrupt` (Ctrl-C) propagated all the way to the interpreter's own top-level handler, printing a raw Python traceback and exiting via the ordinary "unhandled exception" path** (ch. 15: "MUST handle user cancellation gracefully", "MUST avoid treating every interruption as a successful completion", "MUST return an appropriate non-success status when the requested operation was interrupted"). A traceback was never mistaken for success (Python's default uncaught-exception handling already exits 1), so this wasn't a correctness gap, but it also wasn't "graceful," and nothing tested or documented what exit code an interrupted run actually returns. Fixed by adding `run()` (`ast_checks/__main__.py`), a thin process-level wrapper around the existing, still-signal-agnostic `main()`: installs the SIGTERM handler above, calls `main(argv)`, and on `KeyboardInterrupt` (from either signal now) prints a single `Interrupted.` line to stderr and returns `1` — the same non-success code every other incomplete-run outcome in `main()`'s own documented exit-code contract already uses (`0012`), rather than introducing a third, signal-specific value that would expand a contract `0012` deliberately kept coarse. `main()` itself is unchanged and remains signal-agnostic and directly testable exactly as the existing ~40 `test_main_*` tests in `tests/test_orchestrator.py` already exercise it; only `__main__.py` gained the wrapper. Regression tests: `test_run_prints_message_and_returns_one_on_keyboard_interrupt`, `test_run_returns_mains_exit_code_normally` (`tests/test_main.py`).

- **`signal.signal()` itself can fail** — `ValueError` if called outside the main thread, `OSError` in some restricted/sandboxed environments (ch. 15: "MUST handle environments where normal signal behavior is restricted or unavailable", "MUST degrade gracefully when optional signal functionality is unavailable"). `_install_sigterm_handler()` wraps the call in `except ValueError, OSError:` (PEP 758), logging a debug line and continuing rather than raising: Ctrl-C still works via SIGINT's own handler either way, so a failed SIGTERM registration only means one signal fewer is handled gracefully, never a reason to abort the run. Regression test: `test_install_sigterm_handler_degrades_when_signal_signal_unavailable` (`tests/test_main.py`), monkeypatching `signal.signal` to raise and asserting `_install_sigterm_handler()` doesn't propagate it.

## Items judged not applicable, with rationale

- **Ch. 15, "MUST ensure that cancellation does not corrupt shared cache state" / "MUST ensure that cancellation does not leave stale locks indefinitely"**: already true before this audit. `_cache.py`'s `_locked()` holds its `flock` inside a `try/finally` that always calls `fcntl.flock(lock_fp, fcntl.LOCK_UN)`, and `flock` is additionally released by the OS itself whenever the holding file descriptor closes — including on an uncaught `SIGKILL`, which no Python-level handler can ever intercept regardless of what this audit adds. Not changed.
- **Ch. 15, "MUST define whether cancellation stops immediately or after completing the current safe operation"**: a documentation gap, not a code gap — the behavior itself (stop at the next safe point, never mid-write) was already correct by construction of the existing temp-file-then-`replace()` pattern. Fixed by stating it explicitly in `run()`'s own docstring rather than leaving it implicit.
- **Ch. 15, SIGKILL specifically**: no Python-level handler can run for `SIGKILL` under any implementation — the kernel terminates the process before any userspace code gets a chance to run. The only residual risk is a stray, harmless `.tmp` sibling file next to a source file or cache blob (never a corrupted target file, since the temp file is always written and validated before the rename) — the same residual `SIGKILL` risk every tool using this write pattern (`mypy`, `black`, etc.) accepts.
- **Ch. 16, entirely**: every bullet in this chapter (parallel-worker result equivalence, competing modifications between two in-process workers, worker-crash recovery, deadlock avoidance beyond `0013`'s already-timeout-bounded single lock, falling back from parallel to serial execution) describes multi-worker-within-one-invocation scenarios that don't exist in this codebase — confirmed by grep. The one bullet that does apply across separate processes ("ensure that parallel workers cannot corrupt shared cache state") is the same cache-locking concern `0013` already hardened. "MUST make diagnostic ordering deterministic despite parallel execution" is satisfied as a side effect of `main()`'s own existing `sorted()` calls over `unprocessable_files`/`rule_failures`/`all_violations`. "MUST define ownership of each file being modified" was already documented by `0012`.

## Consequences

- `ast_checks/__main__.py` gains `_raise_keyboard_interrupt()`, `_install_sigterm_handler()`, and `run()`; `if __name__ == "__main__":` now calls `sys.exit(run())` instead of `sys.exit(main())`. `main()` itself (`ast_checks/__init__.py`) is completely unchanged — still signal-agnostic, still directly callable from tests and from any future non-CLI embedding without installing a process-wide signal handler as a side effect.
- A SIGTERM sent to the hook process (e.g. by `prek`'s own timeout, or a CI cancellation) now behaves the same as Ctrl-C: a clean `Interrupted.` message on stderr, exit code `1`, and every in-flight file/cache write left either fully committed or fully rolled back — never partial.
- No new exit code was introduced; an interrupted run reuses the same `1` every other incomplete-run outcome already returns, keeping `0012`'s deliberately coarse `0`/`1`/(`2` via `argparse`) contract unchanged.
