# Investigation report: behavioral contract audit, determinism & idempotence (ch. 9, 10)

Detailed findings behind `docs/adr/0025-behavioral-contract-audit-determinism-idempotence.md`. `docs/behavioral_contract.md` chapters **9** (Determinism) and **10** (Idempotence) were checked line-by-line against `_prefilter.py` (`git_grep_filter()`, `batch_filter_files()`), `_orchestrator.py` (`CheckOrchestrator`, `_generate_cache_key()`, `_apply_fixes()`), `_discovery.py`, `_cache.py`, and `_diagnostics.py`'s `report()`, following on from `0002`'s original fix-engine audit (which already spot-checked idempotence ad hoc), `0007`'s parallelism audit, and `0011`'s rule-isolation audit, per issue #40 (blocked by, and sequenced after, #30/`0002`'s fix-engine write-and-validation audit).

## Gaps found and fixed

- **`git_grep_filter()`'s match order depended on `PYTHONHASHSEED`, a value randomized per process by default** (ch. 9: "MUST NOT allow hash-table, process, or thread scheduling order to affect the result"; "MUST produce deterministic diagnostics for identical input, configuration, environment, and tool version"). The successful-match branch built its result by iterating `git_matches`, a plain `set` of path strings parsed from `git grep`'s own stdout, rather than anything insertion-ordered. Reproduced concretely: calling `git_grep_filter()` with 8 real matching files under 5 different `PYTHONHASHSEED` values (0-4) returned 5 different orderings of the identical 8 paths. This was invisible in the one real call path this codebase has — `batch_filter_files()`, the orchestrator's only caller, immediately re-sorts its result (`return sorted(all_matches)`) — but `git_grep_filter()` is itself an exported, independently documented function (`__all__ = ["batch_filter_files", "git_grep_filter"]`, with its own docstring showing direct standalone use), so the underlying nondeterminism was real, not merely theoretical, for any caller that trusts its return order without re-sorting. Fixed by building `matches` from `input_map.items()` — a `dict`, whose iteration order is insertion order (Python's own guarantee since 3.7), matching `filepaths`' own order — instead of iterating the hash-ordered `git_matches` set directly. Regression test: `test_git_grep_filter_match_order_is_independent_of_hash_seed` (`tests/test_prefilter.py`), which spawns three real subprocesses with explicit, different `PYTHONHASHSEED` values against 10 real matching files and asserts all three agree; reliably produces 3 distinct orderings against the pre-fix code (confirmed by temporarily reverting the fix) and 1 against the fixed code.

- **Fix idempotence (ch. 10: "MUST ensure that applying the same fix repeatedly converges to a stable result"; "MUST test fix idempotence explicitly") had never been a committed regression test, only a one-off manual spot-check recorded as prose in `docs/audits/0002`** ("spot-checked by running `--fix` twice over every `tests/fixtures/**/bad/*.py` fixture across all 6 checks: all 33 converged after one pass, none changed on a second pass"). That check was never turned into code, so a future change reintroducing a fix cycle (two checks' fixes toggling the same construct back and forth, or a single check's fix not fully resolving its own violation) would have no test to catch it. Added `test_fix_converges_after_one_pass_across_all_checks` (`tests/test_orchestrator.py`), parametrized over all 33 fixtures currently under `tests/fixtures/**/bad/*.py`, running the full enabled-check set's `--fix` twice in a row over a copy of each and asserting the second pass's file content is byte-identical to the first pass's. Reproduces `0002`'s original finding as a permanent, automatically-run assertion: all 33 fixtures still converge after one pass.

- **Cache-hit/cache-miss equivalence (ch. 9: "MUST ensure that cache hits and cache misses produce equivalent lint results") was only weakly covered.** The existing `test_process_files_second_call_uses_cache` proved a second call actually reads from cache (via a monkeypatched `_check_file` that raises if called), but only compared the resulting violation's `error_code` field, not the full diagnostic. Added `test_cache_hit_and_cache_miss_report_equivalent_violations`, comparing every field a printed diagnostic depends on — `check_id`, `error_code`, `line`, `col`, `message`, `fixable` — between a genuine cache-miss run and a genuine cache-hit run (same proof-of-cache-hit technique), across two checks producing two violations on one file. `fix_data` is deliberately excluded from the comparison: `_cache_violations()`'s own comment already documents that it's never serialized to the cache (it may hold AST nodes), and report-only mode — the only mode that ever reads a cached result, since `process_files()` skips the cache entirely whenever `fix_mode` is set — never consults `fix_data` either, so its absence on a cache-hit violation is already by design, not a gap.

## Already satisfied, confirmed with citations

- **Ch. 9, "MUST define a deterministic ordering for applying multiple fixes"**: already true and already documented by `0003`'s audit (ch. 5 items 11/12, the same underlying fact this ch. 9 item names from a different angle) — fixes are committed per `(check, file)`, each check's own `fix()` writing and atomically replacing the file immediately, in `self.checks`' fixed order (`ALL_CHECKS`, filtered by `--select`/`--ignore`), never batched or reordered.
- **Ch. 9, "MUST define a deterministic ordering for diagnostics"**: `report()` (`_diagnostics.py`) already sorts `unprocessable_files`, `rule_failures`, and `all_violations.items()` via three separate `sorted()` calls before printing anything. Within one file, the violations list itself is ordered by check-declaration order (`ALL_CHECKS`'s fixed order, as filtered by `--select`/`--ignore`) and then by each check's own internal AST-traversal order — not by line/column — which is fully deterministic (confirmed: no set/dict-hash-order dependency feeds into it) but was previously undocumented as an intentional convention rather than incidental. No chapter in `docs/behavioral_contract.md` mandates sorting by line/column specifically, so this is recorded here as the defined ordering rather than changed.
- **Ch. 9, "MUST NOT allow parallel execution to change the final source output"; "MUST NOT allow ... process, or thread scheduling order to affect the result"**: `0007`'s audit already confirmed, by grep, that no threading or multiprocessing exists anywhere in this codebase's own execution — every check runs sequentially within one process. Cross-process concurrency (multiple `prek`-dispatched processes) is bounded by `CacheManager`'s per-file `flock()` (`0010`, `0013`), which this audit re-confirmed is unchanged.
- **Ch. 9, "MUST NOT make output depend on nondeterministic filesystem traversal order unless explicitly documented"**: every filesystem traversal in `src/` is already sorted at its own call site — `_discovery.py`'s `_list_python_files_in_dir()` (`sorted(... resolved_dir.rglob("*.py"))`) and `_cache.py`'s `compute_tree_hash()` (`sorted(root.rglob("*.py"))`) both wrap their own `rglob()` call directly; `batch_filter_files()` returns `sorted(all_matches)`. Grepped for `os.walk`/`os.listdir`/`scandir`/`.glob(`/`.rglob(`/`iterdir` across `src/`: no unsorted traversal exists.
- **Ch. 9, "MUST ensure that running the tool twice on unchanged input produces the same result"**: follows directly from the git-grep fix and the new cache-equivalence test above — a fresh (cache-miss) run followed by an unchanged (cache-hit) run now provably produce identical diagnostics, field-for-field.
- **Ch. 9, "MUST ensure that equivalent execution modes do not silently produce different fixes"**: `process_files()` gates both the cache read and the cache write on `not self.fix_mode`, so `--fix` always recomputes fresh and never reads or writes cache entries — there is no cached/uncached split for fix mode to diverge across. Per-check isolation (`0011`) already ensures a `--select` subset's own fix behavior for the checks it does run is unaffected by which other checks also happen to be enabled in the same invocation.
- **Ch. 9, "MUST make environment-dependent behavior explicit when true determinism is impossible"**: the one genuinely environment-dependent behavior this audit found (`PYTHONHASHSEED`-dependent ordering) is the gap fixed above, and it turned out fully fixable rather than merely something to document as an accepted limitation. The two environment dependencies already known and already documented — cache directory location relative to cwd (`0009`) and interpreter `major.minor` version as part of the cache key (`0011`) — remain the only ones, and both were already explicit before this audit.
- **Ch. 10, "MUST NOT produce an infinite fix loop"; "MUST NOT cause each invocation to make another cosmetic change to the same source indefinitely"; "MUST ensure that a successful fix run does not produce new fixable violations indefinitely"**: all three follow from the same evidence as the idempotence gap fixed above — the new regression test now proves convergence after one pass for every existing fixture, and no check's own `fix()` contains a retry loop or multi-pass logic that could itself run away.
- **Ch. 10, "MUST ensure that the formatter and fixer do not continually fight each other when both are enabled"**: already judged not applicable by `0008` — this hook has no visibility into whether a separately-configured formatter hook (e.g. `ruff format`, run as its own independent pre-commit/prek hook) also touches the same file in the same commit; `pre-commit`/`prek`'s own convention of re-staging and re-running every hook until the working tree stops changing is the actual safety net for that cross-hook interaction, not something this hook could detect or control itself. Re-confirmed unchanged by this audit.

## Items judged not applicable, with rationale

- **Ch. 10, "MUST define behavior when two rules intentionally transform the same construct in opposite directions"**: no such pair exists among this project's 6 checks — each owns a disjoint concern (forbidden variable names, excessive blank lines, redundant `super().__init__()` calls, function naming, redundant assignments, misplaced comments), and none is designed as the deliberate inverse of another. There is no opposite-direction pair to define behavior for.
- **Ch. 10, "MUST detect or prevent fix cycles between rules when possible"**: no in-code cycle-detection mechanism exists, and none was added. Verified instead by (a) the new idempotence test, which exercises all 6 checks running together (not each in isolation) across 33 real fixtures and finds no oscillation, and (b) the structural argument that `_apply_fixes()` always recomputes a check's own violations against the file's current on-disk state immediately before calling that check's `fix()` (audited already by `0002`), so no check within a single invocation can ever act against another's now-stale, already-shifted positions. Building a general-purpose cross-rule cycle detector for a fixed set of 6 checks with no plugin mechanism (`0011`) would be speculative engineering this project's own guidance (`AGENTS.md`) explicitly discourages; "when possible" is read here as best-effort empirical verification, which this audit's new test now provides.

## Consequences

- `src/pre_commit_hooks/_prefilter.py`: `git_grep_filter()`'s successful-match branch now builds `matches` from `input_map.items()` instead of iterating the `git_matches` set directly. No observable behavior change to the real pipeline (`batch_filter_files()`'s own `sorted()` already erased the difference for every existing caller) — only `git_grep_filter()` used directly, outside `batch_filter_files()`, is affected.
- `tests/test_prefilter.py` gains `test_git_grep_filter_match_order_is_independent_of_hash_seed`.
- `tests/test_orchestrator.py` gains `test_fix_converges_after_one_pass_across_all_checks` (parametrized over 33 existing fixtures) and `test_cache_hit_and_cache_miss_report_equivalent_violations`.
- No change to diagnostic ordering, fix-application ordering, or caching behavior for any existing scenario — this audit's code change is confined to the one nondeterminism gap above.
